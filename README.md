# Тестове завдання — Fullstack (Middle) · Fintech / Платежі

Привіт! Це невеликий **запускний** Next.js + PostgreSQL проєкт із внутрішніми
P2P-переказами між рахунками. Код уже написаний і працює на демо — але в ньому є
проблеми, через які в продукті вже траплялися інциденти з грошима.

Ваше завдання — **розібратися, що не так, виправити це і пояснити свої рішення.**

> 🕐 **Орієнтовний час: ~1 година.** Ми не чекаємо ідеального продакшену — нам важливо
> побачити, як ви мислите про код, що працює з грошима.

## Крок 0 — створіть свою копію

> Працюйте над **власною копією**, а не над цим репозиторієм.

1. Угорі цієї сторінки натисніть зелену кнопку **«Use this template» → «Create a new repository»**.
2. У вікні створення:

   - **Visibility — `Public`**.
3. Далі працюйте у **своєму** новому репо (інструкції із запуску — нижче).

> Кнопка «Use this template» доступна, бо це template-репозиторій. Ваша копія —
> незалежний репозиторій (це **не форк**), тож вона не зʼявляється у списках цього репо.

## Стек

- **Next.js (App Router)** + React + TypeScript
- **PostgreSQL** + **Prisma**
- Server Actions для бізнес-логіки

---

## Як запустити (≈ 3 хвилини)

Потрібні: **Node.js 18+** і **Docker**.

```bash
# 1. Підняти Postgres
docker compose up -d

# 2. Налаштувати env
cp .env.example .env

# 3. Залежності
npm install

# 4. Схема БД + клієнт Prisma
npm run db:push

# 5. Тестові дані (3 рахунки)
npm run db:seed

# 6. Старт
npm run dev
```

Відкрийте **http://localhost:3000** — побачите список рахунків і форму переказу.

Після сидингу:

| Рахунок | id | Баланс | userId |
|---|---|---|---|
| Alice | `acc-alice` | 1000 USD | `user-1` *(поточний користувач)* |
| Bob | `acc-bob` | 500 USD | `user-2` |
| Carol | `acc-carol` | 0 EUR | `user-3` |

> «Залогінений» користувач визначається у `src/lib/auth.ts` (зараз це `user-1`, Alice).

Скинути дані будь-коли: `npm run db:reset`.

## Де дивитися

Переказ викликається з **двох** місць (як у реальній системі):

- **`src/app/actions/transfer.ts`** — `transferMoney`, виклик із UI (Server Action).
- **`src/queue/transfer-consumer.ts`** — той самий переказ із **черги** (гарантія
  `at-least-once`), обгорнутий у шар ідемпотентності **`src/lib/idempotency.ts`**.

Допоміжне: `prisma/schema.prisma`, `src/lib/auth.ts`, `src/lib/prisma.ts`.

Спробуйте кілька переказів через UI і подивіться, чи завжди баланси поводяться так,
як мали б. Кілька ідей, що варто перевірити вручну: швидко натиснути «Transfer» двічі
поспіль; переказати дуже велику або незвичну суму; переказати між рахунками з різною
валютою. А далі поставте собі питання: що буде, якщо **те саме повідомлення з черги
прийде двічі одночасно**?

## Завдання

### Частина 1 — Аналіз (≈ 20 хв)

Прочитайте `transferMoney`, схему БД, а також `src/lib/idempotency.ts` і
`src/queue/transfer-consumer.ts`. Знайдіть проблеми та запишіть їх у файл
**`ANSWERS.md`**. Для кожної вкажіть:

- **що не так** (1–2 речення);
- **критичність**: Critical / High / Medium / Low;
- **чим це загрожує** саме платіжній системі.

Якщо проблем кілька — **відсортуйте їх у порядку, у якому ви виправляли б їх першими**,
і коротко поясніть цей порядок. Нас цікавить не лише «що зламано», а **як ви
розставляєте пріоритети**.

> 💡 Шар ідемпотентності `runOnce` *виглядає* правильним. Уважно простежте, що
> станеться, якщо два однакові повідомлення з черги оброблятимуться **одночасно** або
> якщо воркер впаде посеред обробки.

### Частина 2 — Виправлення (≈ 25 хв)

Виправте `transferMoney` так, щоб переказ був **безпечним**. Зокрема подумайте про:

- **атомарність** — переказ має відбутися або повністю, або ніяк;
- **конкурентність** — два одночасні перекази з одного рахунку не повинні «загубити»
  гроші чи піти в мінус (поясніть у `ANSWERS.md`, як саме ви це гарантуєте);
- **коректне представлення грошей**;
- **авторизацію** — переказ дозволено лише з рахунку поточного користувача
  (`userId` беріть з `auth()`);
- **валідацію** входу (сума, валюта, переказ самому собі).

> **«Відтвори і доведи».** Перш ніж/після фіксу гонки чи ідемпотентності — напишіть
> короткий скрипт (наприклад, `scripts/repro.ts`), який **запускає кілька переказів
> паралельно** і показує проблему (баланс «попливе» / подвійне списання). Після фіксу
> той самий скрипт має показати, що проблеми більше немає. Це найважливіша частина: ми
> хочемо побачити, що ви **відтворили й перевірили** поведінку, а не просто «переписали».

Ідемпотентність (`src/lib/idempotency.ts`) виправляти **не обовʼязково за годину** —
але обовʼязково опишіть у `ANSWERS.md`, **у чому там дефект і як ви його полагодили б**
(це обговоримо на дзвінку). Можна змінювати схему Prisma, додавати поля/таблиці — на ваш розсуд.

### Частина 3 — Архітектура та логіка (≈ 15 хв, письмово в `ANSWERS.md`)

Нас цікавить **хід ваших міркувань під конкретні обмеження**, а не загальні «best
practices». Дайте відповіді на **будь-які 2 з 3**:

1. **Ідемпотентність консюмера.** Черга дає `at-least-once`, консюмер працює у
   **кількох інстансах** паралельно, а БД — Postgres. Як зробити `runOnce` коректним,
   щоб одночасні дублікати **не** виконали переказ двічі? Що повертати на дубль і що
   буде, якщо воркер впаде **між** переказом і записом ключа?
2. **Баланси на масштабі.** Рахунок мерчанта отримує сотні переказів за секунду,
   історія — десятки мільйонів рядків, а звіти читаються з **репліки**. Як тримати
   баланс консистентним і швидким для читання? Чи варто переходити на ledger? Як ви
   виявите й виправите розходження (reconciliation)?
3. **Зовнішній провайдер.** Частина переказів іде через зовнішній платіжний API, який
   іноді **таймаутить** (незрозуміло, пройшло чи ні). Опишіть стани переказу й потік,
   що **не допускає** ні подвійного списання, ні втрати грошей.

## Що і як надіслати

Усе — у **вашій копії** (зі Кроку 0):

1. Виправлення коду — комітами в `main` або через PR `solution → main` усередині свого
   репо (так зручніше дивитися дифф).
2. Файл **`ANSWERS.md`** з відповідями на Частини 1 і 3 та короткими коментарями до
   ваших змін (2–3 речення «чому саме так»).
3. Скрипт відтворення з Частини 2 (напр. `scripts/repro.ts`) і команда, як його запустити.
4. Надішліть нам **посилання на ваш репозиторій**. Репо публічне — додаткового доступу
   не потрібно.

> 🗣️ **Вирішальний етап — живий технічний дзвінок (~30–40 хв).** Тестове саме по собі
> ми використовуємо як основу для розмови, а не як остаточну оцінку. На дзвінку ми:
> попросимо **пояснити кожне ваше рішення**, **змінимо умову на льоту** («а якщо…?»),
> попросимо **показати, де у вашому коді ще може лишитися гонка**, і розберемо
> архітектуру з Частини 3. Користуватися довідниками/ШІ під час тестового не заборонено —
> але все, що ви здали, треба буде **захистити вживу своїми словами**. Те, що неможливо
> пояснити, не зараховується.
